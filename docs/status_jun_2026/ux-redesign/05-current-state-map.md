# 05 · Mapa del estado actual del frontend

## Design system & theming

The design system is philosophically sound but functionally inert. Tokens for categorical color (--chart-1..5, --avatar-1..5, --success/warning/info) are comprehensively defined in oklch with dark-mode parity and AA verified, yet remain 95% unconsumed in the feature code. The codebase defaults to raw Tailwind palette (emerald/amber/sky/rose/indigo) hardcoded with hand-written dark: variants, breaking token discipline. Motion exists only in radix primitives (slide-in/zoom via tw-animate-css, shadow-md/lg on selects/dropdowns) with zero signature interactions on the main surfaces (cards, buttons, empty states, transitions between sections). The result reads flat because: (1) categorical colors are token-defined but never applied to pillar cards/activity feed/pillar icons, so the five-hue identity system sits unused; (2) neutral warmth (bone/dark-gray oklch tokens) successfully softens the harsh black/white, but is applied uniformly—no elevation, depth layering, or surface hierarchy via shadow/tone shift; (3) status tokens (success/warning/info) are wired into StatusBadge and Badge but hardcoded Tailwind colors dominate the dashboard, asset history, settings permissions, and BYOI banner; (4) motion is delegated entirely to Radix primitives and skeleton pulses—no CSS or JS microinteractions on user actions, empty/success states, section reveals, or affordance highlights. The warm, disciplined token palette is there; the problem is non-use and absence of elevation/motion strategy.

**Oportunidades:**
- **Activate categorical color system**: Assign --chart-2..5 (teal/green/amber/rose) to the four pillar card headers (Assets/Access/Consumables/Knowledge) via a pillar-color() helper; tint the icon chip backgrounds to match; update activity feed to use --chart-* instead of hardcoded emerald/sky/amber; wire asset-status-badge, article-status, and user-status to map their state enums to --success/--warning/--info/--destructive instead of raw Tailwind palette. One-time refactor, then pillar identity is automatic everywhere.
- **Establish elevation / depth via layer tokens**: Define shadow + ring intensity pairs (e.g., --surface-1 with shadow-md, --surface-2 with shadow-lg, --surface-3 with no shadow but slightly lighter muted bg) to stratify card, detail-panel, and sticky-bar layers—modern design systems have a depth scale. Use shadow-[0_1px_3px_rgba(0,0,0,0.1)] (subtle) for cards, shadow-lg for floating elements, no shadow for muted-bg surfaces. Apply ring-foreground/15 (slightly more visible than /10) for active surfaces.
- **Motion signature moments**: Add CSS keyframes via tw-animate-css or @layer utilities for (1) a gentle in-scale (scale-95 → scale-100, opacity 0 → 1) on cards entering viewport; (2) a soft pulse (opacity 80 → 100) on status badges to draw attention without alarm; (3) a slide-up on success/empty states; (4) a subtle underline-expand on hover for link-like surfaces (breakdown rows in pillar cards); (5) a scale-down + fade on button active press. Respect prefers-reduced-motion.
- **Finish avatar color coverage**: Wire avatarColorFor() to every user display (asset owner, access grantee, assignee list, settings page user list). Audit all places that render users/people: users/_components, assets detail owner chip, applications detail grantee rows, sidebar user menu. Cascade so one seed → one color across all screens (identity consistency).
- **Pillar-scoped color inheritance**: Create a --pillar CSS variable (set by parent route e.g. `/assets -> --pillar: var(--chart-1)`) that downstream components (PageHeader, icon chips, accent underlines, status badges within that pillar) can inherit via currentColor or explicit var(--pillar). No copy-paste color per pillar.
- **Surface stratification by intent**: Differentiate list row (no shadow, ring-foreground/10), list card (shadow-sm, ring-foreground/10, hover:shadow-md), detail panel (shadow-md, ring-foreground/15), and sticky batch bar (shadow-lg, bg-background/95 + backdrop-blur). Let shadow + ring intensity communicate hierarchy instead of opacity tricks.
- **Typography refinement**: Expand the type scale with a semantic --text-* variable set (--text-display, --text-lg, --text-base, --text-sm, --text-xs) bound to specific font-size + line-height + letter-spacing combos so 'large' numbers (metric counts, totals) and 'small' labels (breakdowns) scale as named, not pixel-guessed. Add --font-weight-* aliases for consistency.
- **Micro-affordances**: Add hover:opacity-80 / active:opacity-70 to clickable divs; add focus-visible:ring-2 focus-visible:ring-ring to all interactive surfaces; add data-[slot=active]:text-primary to sidebar/nav items when active to align with button active states; add a subtle glitch-free border-color transition (duration-150) on inputs/textareas.
- **Empty/success state visuals**: Create a @component EmptyStateIcon that scales an icon 3x on a muted-bg circle + optional text. Add a success checkmark animation (strokeDasharray SVG or tw-animate-css scale + rotate). Make empty/error/success states feel like moments, not afterthoughts.
- **Soft focus/blur layers**: Apply backdrop-blur-xs to modal overlays and sticky batch bar already (good). Add a semi-transparent cover to the background when a sheet/dialog is open (dark mode friendly via rgba), so floating content reads as elevated.

## Components, primitives & screen inventory

lazyit's UI is built on disciplined, token-driven primitives over a warm-neutral oklch design system (ADR-0011). The shell (sidebar + topbar + breadcrumb) is clean but minimal—no visual hierarchy, motion, or depth. Primitives (button, badge, status-badge, card, avatar, dialog, table, detail-panel) are well-composed and consistent, leveraging Tailwind v4 tokens for color coordination. However, the app today feels visually neutral: category hues (--chart-1 through --chart-5, anchored on deep indigo + teal/green/amber/rose) are defined but underused; cards, panels and buttons are flat with thin rings (ring-1 ring-foreground/10); empty/loading/error states are utilitarian; no motion (tw-animate-css is installed but only animate-spin and animate-pulse appear); and interaction feedback is reduced to hover opacity/bg changes. List pages (dashboard, applications, assets, consumables, locations, users, kb) repeat a unified pattern (ResourceTable + filter bar + empty/error states + pagination), while detail pages use DetailPanel + DetailField for read-only sections. The foundation is solid and AA-compliant—evolution can add depth, pillar-specific color, microinteractions, and delightful moments without breaking token discipline.

**Oportunidades:**
- Pilot a per-pillar color identity: Inventory (teal --chart-2), Access (already violet in activity but should be indigo --chart-1), Knowledge (green --chart-3), Manage (amber --chart-4). Recolor nav section headings, list page headers, detail panel accents, and dashboard pillar cards to reflect their area; avatar palette stays canonical.
- Add depth via card elevation: introduce subtle box-shadow (e.g. shadow-xs, shadow-sm) and/or increase ring opacity on hover; vary card sizes (sm/default/lg) with corresponding padding/gap scaling.
- Introduce microinteractions: focus states with ring + scale, button press feedback (active:scale-95), smooth hover transitions on links and rows (transition-all), skeleton pulse intensity variation, toast slide-in/out animation (Sonner already supports this).
- Redesign empty/error states as delightful illustrations or fuller layouts (larger icons, softer gradients, more breathing room, action CTA prominence).
- Add badge/status richness: semantic status badges already solid-fill (good); introduce a 'subtle' variant (tinted background, darker foreground) for less critical contexts; category badges in lists could adopt a light pillar tone.
- Signature moment on dashboard: the 'Needs attention' zone could use a warmer hue intensity (ring-warning/20 or ring-danger/30), icon backgrounds with a touch of pillar color (not just generic muted), and a subtle scale/rotate transform on attention rows on hover.
- Introduce a sticky 'jump to section' breadcrumb or progress indicator on long detail pages (assets, users).
- Motion on list pagination: smooth page transitions with a fade + slide, loading skeleton stagger (nth-child delays), row hover reveal of actions button.
- Avatar grouping: when multiple avatars stack (grant recipients, asset owners), show a smoother visual join (negative margin, ring coordination) and a richer 'count badge' component.
- Form field elevation: inputs could have a subtle focus-state shadow lift or scale, making form pages feel less flat.
- Dark mode gradient accents: use a subtle radial or linear gradient on card backgrounds (e.g. from card to muted at 5% opacity) to avoid pure flat color.

## Data & contract surface

Lazyit's data layer is well-architected with a complete read/write contract spanning 5 pillars (Inventory, Access, Knowledge, Users, Management). The API uses offset pagination (ADR-0030), cursor-pagination for append-only logs, and a unified cross-pillar activity feed. Endpoints are pure data-access functions; hooks wrap them in TanStack Query with disciplined cache invalidation. The dashboard aggregates counts/summaries (point-in-time snapshots via DashboardSummary) and surfaces a unified recent-activity feed (RecentActivityPage) merging 4 sources: AssetHistory, AssetAssignment, AccessGrant, ConsumableMovement. User detail pages can load asset assignments + access grants (nested under user detail, enabling per-person offboarding workflows). User password resets are IdP-driven; user onboarding/offboarding is soft-delete with explicit restore (ADR-0041). Actor audit fields (assignedById, grantedById, releasedById, revokedById, performedById) are set server-side from Bearer JWT, never from request body (ADR-0023, ADR-0038). Search is cross-entity (5 types: assets, articles, applications, users, locations) with client-side hit limiting. The data contract is locked (no mutations to shared schemas without ADR discussion); hooks invalidate caches atomically (e.g. grant writes invalidate accessGrantKeys, applicationKeys, userKeys together). Soft-delete is lifecycle-only (ADR-0006): releasedAt/revokedAt are closure markers, not security boundaries. Historical reads (activeOnly=false) include all rows; active reads (default) filter by closure markers. Consumable movements are append-only with transactional stock reconciliation; asset/access history is read-only append-only logs (no edit/delete).

**Oportunidades:**
- Add per-user activity history endpoint (GET /users/:id/activity) filtering RecentActivityItem by actorId — would power 'my history' + audit dashboard
- Add cross-entity history filter (GET /dashboard/activity?entityType=asset&entityId=xxx) to surface all changes to a single resource across all 4 activity sources
- Add filterable history by date range (GET /assets/:id/history?from=&to=) beyond cursor pagination — would enable 'what changed this quarter' reports
- Add consumable movement range queries as bulk read (GET /consumables/movements?consumableId=xxx&consumableId=yyy&type=IN&from=&to=) — parallel to the single-consumable ledger
- Add asset/user assignment history endpoint (GET /users/:id/assignments?activeOnly=false) — client already loads it but no filter params; server-side date range would avoid full-page loads
- Add access grant revocation reason / notes in bulk (already have updateAccessGrantNotes, add batch variant) — would improve offboarding audit trail
- Add 'offboarding manifest' endpoint (GET /users/:id/offboarding) returning nested summary: user + all active grants (to revoke) + all active assignments (to release) + articles authored — single read for the offboarding flow
- Add consumable movement reason enum / curated values (currently free-form string) — would enable 'stock out reasons' filter (damaged, sold, consumed, …)
- Add asset/assignment/grant export (CSV/JSON list) with filters and date range — would feed reporting requirements
- Add recent-activity type breakdown (GET /dashboard/activity/stats) returning { created, assigned, released, granted, revoked, stock_in, stock_out }× date range — would power mini-charts on dashboard right-rail without pagination
