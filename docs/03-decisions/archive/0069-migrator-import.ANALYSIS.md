# Pre-ADR Analysis — Migrator Agent (Bulk Import Wizard) → ADR-0069

**Status:** pre-decision review. Grounded in repo code/docs; deviations from the supplied analyses flagged where the code disagrees.
**Locked (do not relitigate):** (a) JSON + CSV; (b) categories/statuses/models conflicts resolved interactively per-conflict (match / create / skip — never wipe, additive only); (c) history import deferred to phase 2 (phase 1 = current-state entities only).

---

## 1. Executive summary

The migrator is not "one feature" — it is **five hard subsystems welded together**: (1) a brand-new untrusted-file parse/coerce layer (no CSV/Excel parser exists in the repo today), (2) a natural-key → FK **reference-resolution engine** that does not exist in any create path, (3) the interactive conflict resolver, (4) a stateful multi-request wizard session (upload → count → map → dry-run → commit) for which there is **no precedent** — the only import precedent (article import) is single-shot fire-and-forget, and (5) a per-row commit loop that must preserve transactional history + actor attribution at scale.

The biggest risks, in order: **(1) a fundamental architectural contradiction** — locked decision (b) requires a human in the loop *during* conflict resolution, but the only async-bulk precedent runs to completion unattended; this forces a **dry-run → persisted resolution-plan → replay-at-commit** shape that the ADR must specify or the build will either block a worker on human input or silently skip interactivity. **(2) User import is categorically different** from every other entity: each row is a synchronous Zitadel write-back that *hard-deletes the local row and 503s on failure* (verified, `users.service.ts:474–533`) — it cannot live in a Prisma transaction, has no idempotent resume, and a `role` column is a mass-assignment ADMIN-escalation vector. **(3) Silent data corruption** from CSV column-shift / encoding mojibake / type-coercion landing in the exact unique keys (email/serial/sku) used for conflict matching. **(4) currentStock cannot be set via the create schema at all** (verified) — phase-1 consumable import has no valid stock sink without a deliberate decision. **(5) Soft-deleted "ghost" rows are invisible to normal reads**, so "create new" silently succeeds-then-409s and the operator is never offered "restore" unless conflict detection uses `includeSoftDeleted`.

Honest unknowns the CEO must close before building: scope (which entities; are users / assignments / grants in phase 1), the file-staging substrate + PII retention, the commit-failure contract, and whether `role`/ownership are importable at all.

## 2. Per-dimension synthesis

### 2.1 Input & arrival
**Key findings.** No CSV/JSON-array parser exists — this is greenfield untrusted-input ingest, not an extension of the markdown-tolerant article path. CSV is "everything-is-a-string and flat": the existing `z.strictObject` Create schemas (strict enums, `z.iso.datetime()`, `z.cuid()` FKs, nested `specs`) **cannot validate raw CSV rows** — a coercion/normalization layer that runs *before* the Create schema is the actual heart of the feature and is currently unspecified. JSON arrives in incompatible shapes (bare array, `{data:[…]}`, NDJSON, single object, nested sub-objects). Naive `buffer.toString('utf-8')` (the article habit) mangles BOM / Latin-1 / UTF-16 Excel exports — corrupting the unique keys used for matching. CSV dialect (delimiter, quoting, embedded newlines), header detection, and ragged rows all break *before mapping starts*. The article transport (base64-the-whole-file into the BullMQ job) does not scale and is wrong for PII held across requests.

**Recommended approach.** Pick the CSV parser **in the ADR** (streaming, RFC-4180-aware, `bom:true`, strict column-count) — per CLAUDE.md #6 pull its current docs via Context7 before pinning a version. Define an **arrival-coercion contract** (trim; treat `''`/whitespace/null-tokens as *absent* so `.optional()`/`.default()` fire — the empty-string-vs-omit distinction is load-bearing for `unit→'units'`, `role→VIEWER`, `isCritical→false`); case-insensitive enum + value-synonym mapping; date coercion to ISO with the detected format shown for confirmation; boolean/number coercion with declared locale. **The coercion layer is a strict PRE-PASS feeding the unchanged Create schema** — never a replacement (it must still run the schema's own `trim`/`toLowerCase`/`default`, or citext key-matching breaks). Single-source the coercion + normalization in `@lazyit/shared` so the web preview and the API commit agree byte-for-byte ("preview lies" is a whole bug class). Phase-1: **CSV + JSON only; reject `.xlsx`** with an actionable "export to CSV UTF-8" message. Always upload multipart (the Express body-parser default 100kb limit, never raised in `main.ts`, would 413 JSON bodies); raise a dedicated `MIGRATOR_MAX_IMPORT_SIZE_MB`; **drop unmapped columns** before validation (strictObject rejects unknown keys → 400 the whole row).

### 2.2 Field mapping & reference resolution
**Key findings.** **No natural-key resolution exists today** — every Create schema takes FKs only as resolved `z.cuid()`/`z.uuid()` ids; import files carry names/emails/SKUs. The mapping model is actually **three layers**, not one: column→field, value→enum (status/role/LocationType/AccessLevel — the highest-frequency silent mis-map), and field→FK-resolution. Reference resolution is structurally ambiguous: `AssetModel.name` is **not unique** (only nullable `sku` is), and person display names are explicitly non-unique — so "match a model/assignee by name" can yield 0/1/N candidates and must **never auto-pick** (wrong owner pollutes the immutable AssetAssignment ledger). Asset **assignment is not a CreateAssetSchema field** — an "assigned to" column needs a separate, separately-validated write. FK resolution carries a **dependency order** (category → model → asset; users → assets → assignments) and a **transitive create chain** (creating a model needs a category). The manager self-FK adds an intra-file forward-reference/cycle problem.

**Recommended approach.** Build reference resolution as a **first-class, value-deduplicated, dependency-ordered subsystem**: collect distinct natural-key values, resolve/create each *once* (cache like the article folder resolver), in order category → model → asset. Per-entity natural keys: category/location/ApplicationCategory by normalized **name**; AssetModel by **sku** (exact, case-sensitive) else *soft* match on case-folded (manufacturer, name) presented as candidates; user/assignee by **email** (the only citext key) — **require email for assignee in phase 1**, treat name-only as unresolvable→skip. **Different normalization per key** (email/username `trim+lowercase`; legajo `trim`-only case-sensitive; sku case-sensitive). Required-no-default fields (`status`, Location `type`, user `email/firstName/lastName`, `name`) must be **mapped-or-given-a-constant before the mapping step can complete**, or 100% of rows fail. Auto-suggest is **suggestion-only, operator-confirmed, never auto-applied**, exact/normalized header matching (no fuzzy in phase 1).

### 2.3 Conflict resolution (categories / statuses / models)
**Key findings.** **"Statuses" are a closed zod enum, not rows** — "create new status" is impossible; this is a *value-map* problem (map incoming string → one of 6 members, or skip the row since status is required), not match/create/skip. **AssetModel match is ambiguous by construction** (only nullable sku is unique) — "create new" can silently mint a true duplicate the DB will accept (no 409). Category name uniqueness is **case-sensitive** (only email is citext) — a case-insensitive match UI + case-sensitive DB lets `Laptops`/`laptops` both go live. **Soft-deleted ghosts force a 4th option** — the locked 3-option menu (match/create/skip) was written before accounting for ghosts; detection must use `includeSoftDeleted:true` and offer **restore-and-match**. **Skip cascade is dangerous**: skipping a category/model must never *silently null* the FK on dependent rows — offer explicit "import without the link (N rows)" vs "skip the N rows", never silent. **Decisions must be memoized per normalized value** (resolve `Laptops` once, not 480×) and **persisted in Postgres** (append-only), not held only in BullMQ job state (eviction window). LocationType and AccessLevel are additional closed enums needing the value-map; ApplicationCategory is a fourth name-conflict entity — generalize the model over `(referencedTable, matchKey, normalizedValue)` from day one.

**Recommended approach.** Two-phase **ANALYZE → APPLY**. ANALYZE builds a distinct-conflict set keyed by `(entity, field, normalizedValue)` with counts + sample rows + dependent-row blast radius; the operator resolves each *once*; the resolution map (immutable once apply starts) persists to a `MigrationConflict` ledger. APPLY consumes it, creating each new category/model **at most once** in the transaction of the first dependent row. **Correction to the supplied analysis:** match-key normalization must be **trim-only** (mirror the verified `z.string().trim()` — the schemas do *not* collapse internal whitespace; collapsing would make imported categories differ from hand-typed ones). Status/LocationType/AccessLevel get value-maps with synonym auto-suggestions (`active→OPERATIONAL`, `retired→RETIRED`). Put `normalizeMatchKey` + synonym tables in `@lazyit/shared`, unit-tested.

### 2.4 Validation, errors & partial failure
**Key findings.** Reuse the exact Create schema per row (`safeParse`) — do not invent a looser import schema (drift breaks soft-delete/uniqueness/normalization invariants). **Date coercion is the single biggest gap**: `z.iso.datetime()` rejects bare dates, US/EU formats, Excel serials, *and even valid RFC-3339 with a numeric offset* (`+02:00`) — so it bites compliant JSON too; every date must be re-emitted via `toISOString()` regardless of source. Boolean/number/enum coercion from strings will mass-reject rows without a coercion layer. **Per-row best-effort (each row its own transaction with its history) is correct** — all-or-nothing (the 200-cap batch model) would discard a 2000-row import on one bad row; the unit of atomicity is **one row + its history (+ any assignment/movement + their history)**. A **mandatory dry-run** runs the full validate+coerce+FK+uniqueness pipeline (with `includeSoftDeleted`) and writes nothing. **Idempotency has no safe natural key** for assets-without-serial, consumables-without-sku, and AccessGrant (no uniqueness at all → silent duplicate grants on re-run). Errors must be reverse-mapped to the **operator's column header + cell value**, aggregated, and PII-free in logs.

**Recommended approach.** New `@lazyit/shared` **row-index-keyed** report schema (the existing `BatchResultSchema` keys by cuid and `ZipImportResult` by path — neither fits new rows that have no id yet; both also lack a field-level "failed-validation" arm). Coercion runs before the strict schema; **omit empty cells** (don't send `''`) so defaults fire. Intra-file dedupe pass on every natural key before commit (catch dup emails/serials within the file). At commit, catch `P2002`/`P2003` per row → record as a per-row failure ("value taken since preview"), never abort. **Correction:** verify `PrismaExceptionFilter` actually maps `P2003`→400 (only `P2002`→409 is confirmed in context) — an unmapped `P2003` would leak a 500 mid-batch. **Open:** consumable opening stock has no create sink (confirmed `currentStock` absent from `CreateConsumableSchema`) — must be a seeding movement or out of scope.

### 2.5 Domain-invariant collisions
**Key findings.** **Auto-tag allocation consumes the counter OUTSIDE the create transaction** (verified ADR-0063 design) — every *failed/rolled-back* row still permanently burns a tag number (gaps accepted but surprising at scale), and a dense-collision bulk import can exhaust the 10,000-retry budget and surface a misleading 409. **currentStock can only move through the append-only ConsumableMovement ledger** — direct write violates ADR-0034; opening stock must be a seeding IN/ADJUSTMENT movement (both require `quantity>=1`, so stock=0 needs no movement; ADJUSTMENT is the more audit-honest "recount to opening figure"). **No persisted audit of the import action itself exists** — the article path records nothing durable about who imported which file. Asset `specs` is unvalidated jsonb (ADR-0007 TODO live) and **model defaults are materialized then overridden** — a CSV import yields *heterogeneously-typed* specs (model-typed keys + file-stringified keys on the same asset). Ghost-row collisions, intra-file dup keys, and AccessGrant's no-uniqueness all produce silent duplication the DB won't catch. **History enum has no IMPORTED value** (the asset-tag backfill already had to reuse `SPECS_CHANGED`).

**Recommended approach.** Validate all of a row's other invariants **before** calling `allocateTag` (so a doomed row never burns a number); make auto-mint an explicit per-import toggle (default OFF) and recommend disabling/seeding the scheme above the imported estate. Import opening stock as a single seeding movement in the same per-row transaction, attributed to the importer. **Add an append-only `ImportRun` ledger** (who/what/when/counts/conflict-summary, file hash not contents) and stamp `{source:'import', importRunId}` into each CREATED event's jsonb payload (no enum migration; matches the `SPECS_CHANGED`-reuse precedent). Phase-1 recommendation: **drop free `specs`/`metadata` mapping** until ADR-0007 per-category validation lands (simplest, safest), or allow it with a documented "unvalidated, CSV stays string" warning. Conflict detection uses `includeSoftDeleted` to offer restore (serves INV-NO-WIPE-RECREATE). Extend the conflict menu to cover serial/assetTag collisions, not just taxonomies.

### 2.6 Async, scale & ops
**Key findings.** **No durable home for wizard state exists** — the article precedent keeps everything in the ephemeral BullMQ job (`removeOnComplete` 3600s / `removeOnFail` 24h); the multi-step migrator needs a real store or the session dies on re-poll/restart. Raw file bytes (PII) have nowhere to live across steps. Parse + count of a large/hostile file must run in the **sandboxed forked child** (the article SEC-002 pattern: `--max-old-space-size`, concurrency 1, in-process, sharing the api 768m `mem_limit`) — but **the child has NO Nest DI** (bare PrismaClient), so it **cannot invoke** `WorkflowTriggerService` / `NotificationsService` / `SearchService`. Per-row creates fire **side-effect storms**: AccessGrant create writes a PENDING WorkflowRun (transactional outbox, ADR-0054) + post-commit webhook + bell; assets fire Meili upsert; consumables fire `low_stock` bells. Commit-failure contract, TOCTOU between dry-run and commit, lock-duration vs long commits, and abandoned-session GC are all unspecified. **Correction (verified):** the cleanup sweep must follow the existing **`setInterval`** sweeper pattern (`notifications-retention.sweeper.ts` — unref'd timer, re-entrancy guard, `NODE_ENV==='test'` skip); `@nestjs/schedule` is **not installed** and a BullMQ repeatable job would be a novel pattern.

**Recommended approach.** Introduce a first-class **`ImportSession`** entity (cuid) + child **`ImportRow`** (autoincrement — thousands/session), **explicitly excluded from `SOFT_DELETABLE_MODELS`** (transient, hard-deletable by GC), with `status` + `expiresAt` (recommend 24h), the resolved **actor captured at upload** and carried into the worker (the child can't re-derive principal). Parse once on upload, persist normalized rows, **discard the raw file** (avoids a blob store). Decide the **commit-failure contract** explicitly: chunked (100–200/tx) + per-row `ImportRow` status + resumable (re-run skips COMMITTED) + **KEEP-PARTIAL** (additive, audit-friendly; rollback is impossible across Zitadel + out-of-tx tag counter). Use `job.updateProgress({phase, processed, total})`; raise `lockDuration` + renew between chunks; commit may be `attempts>1` (idempotent), parse stays `attempts:1` (permanent). **Decide side-effect policy**: suppress workflows/notifications/search during bulk import and emit one post-import summary (recommended), or accept the storm — and resolve where commit runs (in-process with DI + side-effects but no bomb isolation, vs child with isolation but re-implemented outbox and no notifications).

### 2.7 Security & permissions
**Key findings.** **No import permission exists** in the frozen catalog (verified). The article path gates import on `article:write` (same verb as one article) — acceptable for KB, **wrong for a cross-entity migrator** that can mint role-bearing users. **Mass-assignment is the design's core shape**: mapping must funnel through the existing `z.strictObject` schemas (never build a Prisma payload from a dict, never `.passthrough()`) — `externalId` (SEC-006 pre-linking), `id`, `currentStock`, actor/lifecycle columns are closed *by being absent*. `role` **is** in `CreateUserSchema` (verified line 171) — a fuzzy auto-suggest mapping a `role`/`access`/`admin` column mints real ADMINs (DB-first authority, INV-1), compounded by first-user-ADMIN (INV-7) on a fresh instance. Application `url` is already defended by `isSafeApplicationUrl` (SEC-008) **only if routed through the schema**; `specs`/`metadata` jsonb are stored-XSS sinks. PII-at-rest: the staging area holds a staff directory; `getStatus(jobId)` has **no ownership check** (IDOR if copied). CSV-formula injection is a deferred-detonation risk on a future export.

**Recommended approach.** Add a new catalog permission (`import:run`, ADMIN-only by default) **AND** enforce the per-target entity permission at commit — but note `@RequirePermission` gates one static permission per route and the target isn't known until after analyze, so the AND-check must be a **runtime `PermissionResolver` call inside the commit service**, keyed on the chosen entity and each conflict action (`category:write` for create-new, restore-permission for restore). Mappable target fields are **generated from the Create schema key set**; deny `role` auto-suggest, surface "this will create N users, M as ADMIN", default unmapped role to VIEWER (recommend forbidding `role` import entirely in phase 1). Neutralize CSV-formula leading chars on free-text fields; cap `specs` depth/size and confirm the web renderer escapes (route to frontend subagent). Owner-scope the session id; no `@Public`; actor from principal only. Inherit SEC-001/SEC-002 (multer cap + sandboxed child + record-count quota). **No raw rows in logs** — counts + conflict summaries only.

## 4. Error & edge-case catalog (what the importer must handle)

## 5. What ADR-0069 must lock

1. **Wizard architecture:** synchronous validate (auth/type/size/sniff/count) → **async sandboxed parse** (202 + sessionId) → mapping → **synchronous dry-run** (full validate+coerce+resolve+conflict-detect, zero writes) → **persisted resolution plan** → **async chunked commit** (replay plan, re-validate per row). Locks the multi-request, dry-run-mandatory shape.
2. **State model:** new `ImportSession` (cuid) + `ImportRow` (autoincrement), excluded from `SOFT_DELETABLE_MODELS`, hard-deleted by a `setInterval` GC sweeper (NOT `@nestjs/schedule`); raw file parsed-once-and-discarded; actor captured at upload, carried into the DI-less worker.
3. **Mapping model:** explicitly **three layers** — column→field (from the Create-schema key set, auto-suggest confirmable, leftover columns dropped), value→enum (status/Location.type/AccessLevel synonym maps), field→FK-resolution. Required-no-default fields must be mapped-or-defaulted before mapping completes.
4. **Reference resolution:** value-deduplicated, dependency-ordered, `includeSoftDeleted` conflict probe; per-entity natural keys + per-key normalization (email/username lowercase, legajo/sku case-sensitive); never auto-pick on ambiguity.
5. **Conflict model:** four outcomes (match-live / restore-ghost / create-new-when-no-live-match / skip); statuses/enums are value-maps not entity conflicts; **trim-only** match normalization; explicit skip-cascade semantics (never silent FK null); decisions persisted append-only.
6. **Validation:** reuse the exact `z.strictObject` Create schema per row; coercion as a strict pre-pass (omit empty cells); dates always re-emitted via `toISOString()`; new row-index-keyed `@lazyit/shared` report schema with a field-level failure arm.
7. **Commit contract:** per-row transaction (entity + history [+ assignment/movement + their history]); per-row `P2002`/`P2003` handling (no abort); KEEP-PARTIAL + resumable; verify `P2003`→400 mapping; **route every write through the real service.create() path** (never `createMany`) for transactional history + actor + SEC-008 url guard.
8. **Idempotency:** resume-within-session by `ImportRow` status; re-upload dedup only via mapped natural key (else warn); `ImportRun` correlation id for audit/undo.
9. **Async/scale:** sandboxed child + heap cap + record-count quota; separate queue (or accept serialization at concurrency 1); progress reporting; raised lock-duration; **side-effect suppression policy**.
10. **Security:** `import:run` (ADMIN-default) + runtime per-target permission AND-check at commit; mappable fields from schema keys only; `role`/`externalId`/`id`/`currentStock`/actor columns never mappable; owner-scoped session; CSV-formula + specs-jsonb hardening; PII-free logs; staging TTL.
11. **Domain rules:** auto-tag validated-last + toggle-off-default; consumable opening stock as a seeding movement; specs free-mapping deferred; `{source:'import', importRunId}` provenance in CREATED events.
12. **Phasing & docs:** phase 1 = current-state only (no history backfill, reject foreign `createdAt`/`updatedAt`/`deletedAt`); en+es Manual pages in the same change (CLAUDE.md #7); the bilingual UX copy for conflict menus, ghost-restore, partial-success, and the "history is phase-2" warning is in scope.

## 6. Recommended first build slice (thinnest end-to-end vertical)

**Asset import, CSV only, no ownership, no specs mapping, scheme-off.** Rationale: assets are the asset-centric core, pure-DB (no IdP side-effect), exercise the full machinery (FK resolution to Model/Category/Location, the interactive conflict menu, value-map for `status`, required-no-default enforcement, ghost detection) without the user-import IdP minefield or the consumable currentStock invariant question.

The slice:
1. Multipart CSV upload → sandboxed parse → `ImportSession` + `ImportRow` rows, count + detected headers + dialect/encoding shown for confirmation.
2. Field mapping (auto-suggest confirmable; required fields `name`, `status` enforced; `status` value-map; `Model`/`Category`/`Location` columns route to FK resolution by name).
3. Dry-run: per-row create/match-existing/restore-ghost/skip + per-row validation errors, with the distinct category/model/location conflict set resolved once each; persist the resolution plan.
4. Chunked commit through `AssetsService.create()` per row (transactional CREATED history, importer as actor, `importRunId` stamped), per-row outcome report, keep-partial.
5. `import:run` (ADMIN) + runtime `asset:write` + `category:write`/`assetModel:write` checks; en+es Manual page.
