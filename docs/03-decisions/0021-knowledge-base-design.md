# ADR-0021: Knowledge Base design — simple wiki

## Status

accepted — 2026-05-25. One of the three MVP modules ([[vision]]) alongside Inventory ([[asset]])
and Access ([[application]]). Walks back the versioning that the [[article-version]] stub
anticipated. Authorship/visibility rules are split into [[0022-draft-visibility-auth-shim]].

## Context

The Knowledge Base is lazyit's internal-documentation pillar. Its users are **small IT teams
(5–10 people)**: someone writes a note, others read it, someone edits it — done. We explicitly do
**not** want Notion/Confluence scope: no real-time collaboration, no tags, no advanced full-text
search, no per-edit version history. The existing entity stubs ([[article]], [[article-category]],
[[article-version]]) over-anticipated by baking in versioning before the module exists.

We must fix the data shape and the in/out capability set for the MVP **so that everything deferred
can be added later without a destructive migration**.

## Considered options

- **Full wiki now** (versioned, tagged, FTS, collaborative) — matches Confluence; far too much
  surface for a 5–10-person tool, and premature. ❌
- **`Article` + `ArticleVersion` from day one** (what the stubs implied) — versions every edit,
  doubling the write path and read model for a feature nobody has asked for. ❌
- **A single `Article` (markdown) + user-managed `ArticleCategory`, no versioning** — the minimum
  that delivers a usable wiki; every omission is an additive change later. ✅ *(chosen)*

## Decision

- **`Article`** holds the canonical document: `slug` (unique, short — `[a-z0-9-]`, ≤60 chars,
  autogenerated from the title but **editable**), `title`, `content` (markdown), optional
  `excerpt` (≤280), `status` (`DRAFT`/`PUBLISHED`), required `categoryId` and `authorId`, optional
  `lastEditedById`, `publishedAt` (set on the **first** publish, **never cleared**), and a flexible
  `metadata` jsonb. Soft-deletable ([[0006-soft-delete-and-auditing]]).
- **`ArticleCategory`** is user-managed exactly like [[asset-category]]: `name` (unique), optional
  `description`, `icon` (a heroicon name, free string), and `order` (for sidebar sorting). Seeded
  with an initial, non-special set.
- **No versioning, no tags, no advanced FTS.** Listing search is a basic case-insensitive `LIKE`
  over `title` and `excerpt` only (not `content`).
- **`status` lifecycle:** `DRAFT` is author-private; `PUBLISHED` is team-visible. Transitions go
  through dedicated `POST /articles/:id/publish` and `/unpublish` endpoints (not `PATCH`, which
  never touches `status`). The visibility + authorship rules are in
  [[0022-draft-visibility-auth-shim]].
- **File import** (`POST /articles/import`, multipart) supports `.md`, `.txt` and `.docx`; it
  extracts **markdown only** and does **not** store the original file. `.docx` is converted with
  `mammoth` (pure JS, no external binaries). Max size via `MAX_IMPORT_SIZE_MB` (default 5).
- **`metadata` jsonb** mirrors the `Asset.specs` debt ([[0007-flexible-asset-specs-jsonb]]):
  `z.record(z.string(), z.unknown())`, **no dynamic validation yet** — documented debt, not a gap.

## Consequences

- **Positive:** the smallest thing that works; reads/writes are plain CRUD over one table plus a
  reference table.
- **Append-improvable (the key property):** every deferred feature is a **non-destructive**
  addition — versioning = a new `ArticleVersion` table with an FK from `Article` ([[article-version]]
  stays ⚪ planned); tags = a join table; FTS = a Postgres `tsvector` column + GIN index. None
  reshape `Article`.
- **Trade-offs:** no edit history (an overwrite loses the previous body until versioning lands);
  search is substring-only and excludes `content`; `metadata` is unvalidated (same debt as `specs`).
- **Deferred import formats:** `.pdf` (ugly text extraction), `.html` (→ `turndown`), `.odt`/`.rtf`
  — documented, **not** built. No `pandoc` / external binaries, ever.

## Open questions

- **Export** — download an article as `.md` / `.pdf`? (not decided)
- **Slug auto-suffixing** — on collision, auto-increment `-2`, `-3`? Today a collision is a `409`
  and the caller picks another slug. (not decided)
- **Per-category `metadata` schema** — same trajectory as the `specs` debt
  ([[0007-flexible-asset-specs-jsonb]]).
