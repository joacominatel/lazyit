# Internationalization (i18n) — how to translate a section

This is the **operating manual for the per-section translation fan-out**. The foundation
(next-intl, cookie-mode, en + es) shipped in Phase 0 ([[0051-i18n-next-intl|ADR-0051]]). Your job
as a section agent: take one product area (e.g. Assets), move its hardcoded strings into the
catalogs, and wire its components to read them — **mechanically**, following the rules below.
Read this top to bottom before touching code.

> **TL;DR** — Each top-level namespace is its own file: `apps/web/messages/{en,es}/<area>.json`.
> The file's content **is** that namespace's subtree (no wrapping object) — keys are referenced
> `<subarea>.<key>` via `useTranslations('<area>')` in Client Components and `await
> getTranslations('<area>')` in Server Components. **Every key must exist in BOTH
> `en/<area>.json` and `es/<area>.json`.** Don't translate data (IDs, enums, route paths, API
> fields, user input). You edit **only your area's two files** — never the `_all.ts` barrels.

## Where everything lives

| Thing | Path |
| --- | --- |
| Locale config (locales, default, cookie name, labels, `isLocale`) | `apps/web/i18n/config.ts` |
| Per-request config (reads the `NEXT_LOCALE` cookie) | `apps/web/i18n/request.ts` |
| `setLocale` server action (writes the cookie) | `apps/web/i18n/actions.ts` |
| **English catalog (per area)** | `apps/web/messages/en/<area>.json` (one file per namespace) |
| **Spanish catalog (per area)** | `apps/web/messages/es/<area>.json` (one file per namespace) |
| Per-locale barrels (FOUNDATION — don't touch) | `apps/web/messages/{en,es}/_all.ts` |
| Locale switcher (topbar) | `apps/web/components/locale-switcher.tsx` |
| Worked example (chrome wired through i18n) | `apps/web/components/sidebar-nav.tsx` |

You will normally touch **only your area's two files** — `messages/en/<area>.json` and
`messages/es/<area>.json` — and the components inside your section's folder (e.g.
`app/(app)/assets/**`). **Do not** touch `i18n/*.ts`, the provider, the root layout,
`next.config.ts`, or the `messages/{en,es}/_all.ts` barrels — that is the foundation; if you think
it needs changing, 🚨 stop and escalate. If a visible string lives in a **shared** component
outside your folder (e.g. `apps/web/components/*.tsx`), leave it — the `shared` pass owns it.

## The namespace scheme: `<area>.<subarea>.<key>`

The catalog is **split one file per top-level namespace** under `messages/<locale>/<area>.json`.
**The file's content is the namespace's subtree directly — no wrapping object.** So
`messages/en/assets.json` is `{ "list": { … }, "form": { … } }` (NOT `{ "assets": { … } }`);
the `assets` key is supplied by the barrel. Phase 0 created these files (sections start as empty
`{}` waiting for you):

```
common      ← seeded full (Save/Cancel/Delete/Edit/Create/Loading/…). SHARED — reuse, don't re-add.
nav         ← seeded full (the sidebar labels).
shared      ← app-level shared components + chrome (owned by the `shared` pass).
dashboard   assets   applications   consumables   kb   users   locations   settings   reports   auth
```

Rules:

1. **One top-level namespace per area.** Put Assets strings under `assets`, Users under `users`,
   etc. Match the placeholder namespace already in the catalog. Don't invent a new top-level key
   without a reason.
2. **`<subarea>` groups within an area** — e.g. `assets.list.*`, `assets.form.*`,
   `assets.detail.*`, `assets.empty.*`. Use subareas to mirror the UI structure (table, form,
   detail, dialogs, empty-state). One level of subarea is usually enough; nest a second only if
   it genuinely clarifies.
3. **Keys are `camelCase`** and describe the *meaning*, not the English text — `addAsset`, not
   `addAssetButton` or `Add Asset`. Good: `assets.form.serialNumberLabel`,
   `assets.list.emptyTitle`, `assets.detail.assignOwner`.
4. **Reuse `common` for generic UI verbs** (Save, Cancel, Delete, Edit, Loading, Search, …)
   instead of re-adding them under your area. Pull them with `useTranslations('common')` (you can
   call the hook more than once with different namespaces, or use the root `useTranslations()` and
   dotted keys — see below).

### Example shape

Each area file holds the namespace subtree **directly** (the `assets` wrapper is added by the
barrel, so it is NOT in the file):

```jsonc
// messages/en/assets.json
{
  "list": {
    "title": "Assets",
    "emptyTitle": "No assets yet",
    "emptyBody": "Add your first asset to start tracking inventory."
  },
  "form": {
    "serialNumberLabel": "Serial number",
    "save": "Save asset"
  }
}
```

```jsonc
// messages/es/assets.json  — SAME keys, translated values
{
  "list": {
    "title": "Activos",
    "emptyTitle": "Aún no hay activos",
    "emptyBody": "Agrega tu primer activo para empezar a llevar el inventario."
  },
  "form": {
    "serialNumberLabel": "Número de serie",
    "save": "Guardar activo"
  }
}
```

Generic verbs (Save/Cancel/…) already live in `messages/{en,es}/common.json` — reuse them with
`useTranslations('common')`, don't re-add them to your area.

## How to read translations in code

### Client Components (`"use client"`) → `useTranslations`

```tsx
"use client";
import { useTranslations } from "next-intl";

export function AssetsTable() {
  const t = useTranslations("assets"); // namespace-scoped
  return <h1>{t("list.title")}</h1>;   // resolves assets.list.title
}
```

Need `common` too? Call the hook again, or scope it differently:

```tsx
const t = useTranslations("assets");
const tc = useTranslations("common");
// ...
<button>{tc("save")}</button>
```

### Server Components / Server Actions → `getTranslations`

```tsx
import { getTranslations } from "next-intl/server";

export default async function AssetDetailPage() {
  const t = await getTranslations("assets");
  return <h1>{t("detail.title")}</h1>;
}
```

`getTranslations` is **async** — `await` it. It works in Server Components, `generateMetadata`,
Route Handlers and Server Actions. `useTranslations` is for Client Components only (it's a hook).

### Picking which one

- Component has `"use client"` at the top → `useTranslations`.
- It's a server component / page / layout / action (no `"use client"`) → `await getTranslations`.
- If a string lives in a server component but is only rendered by a client child, prefer
  translating it in the **client** component that owns the markup, to keep the boundary clean.

## ICU: interpolation, plurals, rich text

next-intl uses **ICU message syntax**. Never build sentences with string concatenation.

**Interpolation** — pass values as the second arg:

```jsonc
{ "assets": { "greeting": "Welcome back, {name}" } }
```
```tsx
t("greeting", { name: user.name });
```

**Plurals** — use the `plural` ICU form (handles 0/1/many per-locale):

```jsonc
{ "assets": { "count": "{count, plural, =0 {No assets} one {# asset} other {# assets}} " } }
```
```tsx
t("count", { count: assets.length });
```

`#` prints the number; `one`/`other` are the CLDR plural categories (English uses `one`/`other`;
Spanish too — but let ICU decide, don't hardcode). Use `=0` for an explicit zero phrasing.

**Numbers / dates** — prefer `t("…", { n: value })` with ICU `number`/`date` skeletons, or
next-intl's `useFormatter()` for standalone formatting. Don't format dates by hand per locale.

## Time zone (issue #548)

next-intl needs an explicit **global `timeZone`** for any date/time formatting that includes a
time part (the shared `long` preset adds `hour`/`minute`). Without one it falls back to the
runtime zone and emits an `ENVIRONMENT_FALLBACK` warning — and the server (the container's zone)
and the client (the browser's zone) can disagree, producing a hydration mismatch.

lazyit is single-org self-hosted with **no per-user time zone in v1**, so we configure ONE
instance-wide display zone:

- **`DEFAULT_TIME_ZONE`** in `apps/web/i18n/config.ts` reads
  `process.env.NEXT_PUBLIC_DEFAULT_TIME_ZONE`, defaulting to **`UTC`**. Set it to any IANA zone
  name (e.g. `America/Argentina/Buenos_Aires`) in `apps/web/.env`.
- The `NEXT_PUBLIC_` prefix is load-bearing: it makes the value readable on **both** the server
  and the client, so the same zone flows to `i18n/request.ts` (server `getRequestConfig`) and to
  `NextIntlClientProvider` (client, `app/providers.tsx`).
- **Server and client MUST read the same constant** — both import `DEFAULT_TIME_ZONE`. Don't
  hardcode a zone in one place and read the env var in the other, or timestamps rendered on the
  server won't match what the client re-renders (hydration mismatch). There is no per-user zone
  and no UI picker yet — changing the instance zone is an env-var change.

## The parity rule (NON-NEGOTIABLE)

**Every key you add to `en.json` MUST exist in `es.json`, and vice-versa.** `en` is the fallback,
so a missing `es` key silently renders English — that is a **defect**, not a feature. Before you
finish:

1. Add the key to **both** files in the same commit-ready change.
2. Sanity-check your area's two files have the same shape. A quick structural diff with Bun
   (replace `assets` with your area):

   ```sh
   # from apps/web — prints keys present in en but missing in es (and vice-versa)
   bun -e 'const a="assets"; const f=(o,p="")=>Object.entries(o).flatMap(([k,v])=>v&&typeof v=="object"?f(v,p+k+"."):[p+k]); \
     const en=new Set(f(require(`./messages/en/${a}.json`))), es=new Set(f(require(`./messages/es/${a}.json`))); \
     const only=(x,y)=>[...x].filter(z=>!y.has(z)); \
     console.log("en-only:",only(en,es)); console.log("es-only:",only(es,en));'
   ```

   Both arrays must be empty. (The orchestrator runs the same check across **all** area files
   before the PR.)

## What NOT to translate

Translate **UI copy** only. Leave as-is (these are data/identifiers, not user-facing prose):

- Entity IDs, slugs, cuids/uuids.
- **Enum/status values** that the API returns or expects (e.g. asset status codes, role names,
  permission strings) — translate the *display label* via a lookup if needed, but never change
  the value sent to/from the API.
- Route paths and `href`s, API field names, query keys.
- User-entered data (asset names, ticket bodies, articles, emails, locations the user typed).
- Code, console logs, error codes meant for operators.

When in doubt: if changing the string could break a comparison, an API call, a URL or a stored
value, it's **data** — don't translate it.

## Definition of done for a section

- [ ] Every visible hardcoded string in your section is moved to `assets.*` (your area) or reuses
      `common.*`.
- [ ] Keys exist in **both** `en.json` and `es.json` (parity check passes — empty diff).
- [ ] Client components use `useTranslations`; server components/actions use `getTranslations`.
- [ ] Plurals/interpolation use ICU, not concatenation.
- [ ] No data/identifiers were translated.
- [ ] `cd apps/web && bunx tsc --noEmit` is clean and `bun run build` is green.
- [ ] You did **not** edit the foundation (`i18n/*.ts`, provider, root layout, `next.config.ts`).
